# Product

## Register

brand

## Users

DAML developers learning the language from scratch. They come from Solidity, Rust, or traditional programming backgrounds. They want to see working code, understand patterns, and get back to building. Reading happens in focused sessions at a desk, often with the docs in one window and their editor in another.

## Product Purpose

An educational reference for learning DAML smart contracts through practical, runnable examples. The site exists to answer one question faster than anything else: "How do I write this in DAML?" Success means a developer finds the pattern they need, copies the code, and adapts it without leaving the page.

Modeled after Solidity By Example — a site where every example earns its place and nothing distracts from the code.

## Brand Personality

Clean, Technical, Precise.

Voice is direct and confident. No hand-holding, no fluff, no "let's learn together" warmth. The language of a senior engineer showing a junior how its done: exact, minimal, matter-of-fact.

## Anti-references

- Medium / blog-style layouts with hero images, pull quotes, and narrative prose. This is documentation, not storytelling.
- Docusaurus and Nextra defaults: generic sidebar-in-card, template-looking documentation.
- Corporate SaaS marketing pages (Stripe-style landing pages are a reference, but the doc pages specifically are what Stripe does well, not the marketing).
- Anything that makes code feel secondary to the chrome around it.

## Design Principles

1. **Code is the hero.** Content serves the code examples, not the reverse. Layout, spacing, and typography exist to make code blocks the most readable element on every page.

2. **Zero ornament.** Every pixel earns its purpose. No gradient accents, no decorative illustrations, no hover animations that dont communicate state. If it can be removed, remove it.

3. **Developer respect.** Trust the reader to handle density. Sidebars list everything, tables of contents show all headings, code blocks show full files. No progressive disclosure, no accordions, no "show more" truncation.

4. **Consistency as trust.** The same heading hierarchy, the same spacing rhythm, the same code block treatment, on every page. Predictability means the reader never thinks about the interface.

5. **Off the shelf.** Must not look like a generated documentation template. Avoid the Docusaurus/Nextra family resemblance. The layout is custom; the proportions are deliberate; the site has a point of view.

## Accessibility & Inclusion

Standard best practices: reasonable contrast ratios, semantic HTML, keyboard navigable sidebar, proper heading hierarchy, focus indicators. No strict certification target but no known barriers.
